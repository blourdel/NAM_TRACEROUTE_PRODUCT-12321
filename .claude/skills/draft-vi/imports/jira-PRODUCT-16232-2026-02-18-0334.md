# PRODUCT-16232 - Segments definition sync and storage drift

**Status:** Open
**Type:** Platform/Data integrity
**Assignee:** Siamak Nezhadi
**Execution Assignee:** Stefan Wolfsteiner
**Owning Program:** Platform Apps
**Labels:** segments
**Created:** 2026-02-02
**Updated:** 2026-02-17

## Short Abstract / Blogline

Fix Segment persistence drift across Dynatrace by ensuring saved/embedded segment selections (Recents/Pinned/Default, Dashboards, Notebooks, URLs) don't become orphaned, stale, or incompatible when Segment names and/or definitions evolve or get deleted.

## Customer Zero

* Notebooks
* Dashboards
* Workflows

## Target Audience

* SREs / performance engineers using Segments for day-to-day filtering and investigations
* Dashboard authors/consumers relying on default filters and shared links

## WHY / Problem Journey

We currently have no single "source of truth" for a complete, reusable Segment selection over time (segment definition + variables + selected state). Each consuming surface persists its own snapshot payload, and these snapshots are not synchronized when segments are deleted or changed.

This creates time-based drift: the longer a saved selection lives, the more likely it becomes inaccurate, misleading, or broken. The impact is amplified because the same segment can be persisted in multiple places, each with its own lifecycle and update behavior.

### Where It Hurts

#### 1) Orphaned References

* What happens: A segment is persisted (recent/favorite/default/link). Later it's deleted or becomes unavailable (permissions/space changes). The stored reference/snapshot can't be resolved.
* Where it shows up: Recents/pinned/default Segments in SegmentSelector across the platform, Dashboards, Notebooks, shared URLs.
* User symptoms: broken/misleading entries (e.g., odd identifiers), apply fails or does nothing, "why is this still here?" questions.

#### 2) Stale Selections

* What happens: Persisted selections replay old variable values (especially arrays that are dynamic/contextual). Over time those values drift from reality.
* Where it shows up: Workflows, Dashboard defaults, URLs, reopened Notebooks.
* User symptoms: silent wrong results (missing new items / still filtering on removed ones), inconsistent outcomes across users or over time.

#### 3) Incompatibility After Evolution

* What happens: Segment definitions/variables evolve (bugfixes, renames, added/removed fields). Older persisted payloads become incompatible with what consumers expect.
* Where it shows up: Workflows/Dashboards/Notebooks/URLs failing to load/apply.
* User symptoms: either silent drift (old/broken fields persist forever) or hard failures (selector/page/query errors), forcing manual rework and making safe iteration risky.

## Functional Requirements / Solution Journey

* Provide a consistent way for any consumer app to persist a segment selection so it can be safely re-used later.
* On load/apply, consumer apps must be able to detect whether a stored selection still matches the current segment definition expectations and:
  * auto-update it when possible, or
  * show a localized, actionable error state when not possible.
* Deleted/unavailable segment references must be handled consistently (not shown as valid; removable).
* Support refresh behavior for dynamic variable values (especially arrays) so reuse doesn't replay stale values.

## Use Cases

1. **Recents/pinned cleanup:** When a segment is deleted/inaccessible, it must not appear as a valid recent/favorite.
2. **Dynamic values stay current:** Reusing a saved selection reflects current valid values (or prompts if prior values no longer exist).
3. **Workflows/Dashboards/Notebooks/URLs survive changes:** Stored selections don't break views/queries when segment variables evolve; users either get auto-updated state or a clear recovery path.

## Non-Functional Requirements

* **Reliability:** Persisted segment selections must not cause page-breaking failures.
* **Backward compatibility:** Best-effort handling of existing stored selections (URLs/docs/appstate).
* **Performance:** Minimal additional latency on load/apply; must be measurable.
* **Usability:** Error states must be understandable and provide clear actions.

## Acceptance Criteria

1. Deleted/inaccessible segments are not rendered as valid in Recents/Pinned, be clearly indicated as invalid and can be removed/cleared by the user.
2. Persisted segment selections used by Workflows, Notebooks and Dashboards or other apps do not cause load/query failures after segment definition changes; they are either auto-updated or surfaced as a localized error.
3. Consumer apps can determine whether a stored segment selection is still compatible with the current segment definition expectations.
4. If compatibility issues are auto-fixable, the system auto-updates the stored selection without user intervention.
5. If not auto-fixable, the segment selector shows an actionable error state (e.g. clear selection + choose different segment/value).
6. For variables marked as dynamic (or governed by refresh rules), reuse does not replay frozen array values; stale/invalid values are refreshed or surfaced via AC5.

## Out of Scope

* Full redesign of segments governance (ownership/sharing/spaces) beyond persistence correctness
* Bulk migration tooling for all historical documents/links across tenants

## Success Metrics

* Reduction in dashboard/notebook incidents and support tickets caused by persisted segment breakage
* Auto-update success rate for outdated selections (and decline in user-facing error states over time)
* Reduction in "deleted segment still in recents/favorites" occurrences
* Drop in query failures attributable to segment selection incompatibilities
